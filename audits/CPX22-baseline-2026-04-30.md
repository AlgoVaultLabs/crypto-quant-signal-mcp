# CPX22 Baseline CPU Audit — AlgoVault-MCP (`204.168.185.24`)

**Audit window**: 2026-04-30 11:30–12:30 UTC (post-SHADOW-SEED-W1-recovery state, mid-day European business hours)
**Box**: Hetzner CPX22 (2 vCPU AMD EPYC-Genoa @ 2.0 GHz, 3.7 GiB RAM, 75 GB NVMe), Helsinki
**Status**: Read-only audit — zero production state mutations.
**Trigger**: Phase 0 of SHADOW-SEED-W1 measured load avg **1.74 (5m) / 1.98 (15m) = 87-99% normalized** BEFORE shadow-seed went live. Audit answers: what's eating that baseline, can the 1m cron return to `* * * * *`, and is CPX31 upgrade justified.

---

## TL;DR (Headline Findings)

1. **Postgres is ~80% of the baseline CPU**. Sustained 40.73% measured at peak snapshot (14 ESTABLISHED connections, 4-conn pool against the postgres container, ~7+ concurrent active queries per second). On a 2-vCPU box that's already ~1.0 of total CPU capacity before any cron fires.

2. **The dominant query is `SELECT funding_rate FROM funding_history WHERE coin = $1 AND recorded_at >= $2 ORDER BY ...`** — fired by `getFundingZScore()` from inside every `getTradeSignal()` call (per-coin, per-call, no in-memory cache). With 250+ trade signals/min generated by cron + grid + user traffic, this query runs ~150-200×/sec sustained. **Each query is fast (0.13ms execution, index-served)** — the problem is volume × frequency.

3. **funding_history table is 565 MB** (heap 248 MB + indexes 316 MB) holding **5.2 M rows** spanning 22 days at ~1-row-per-7-seconds-per-coin density. Index size is larger than heap; suggests no recent reindex. Last_autovacuum: NULL (autovacuum never analyzed the table — stats were `n_live_tup=8801` while real count was 5.2M).

4. **Stats are catastrophically stale across all tables** (all `last_autoanalyze` and `last_autovacuum` columns NULL). The query planner is picking plans with bad row-count estimates; once `ANALYZE` runs, plans should improve.

5. **One OOM-kill event in dmesg** at 2026-04-30 10:54:14 UTC — the SHADOW-SEED-W1 cron-load incident. Pre-incident: the box has been up 24 days clean (no OOMs in the rotated log windows reviewed).

6. **Logs are growing unbounded** — 5×seed-*.log at 13-32 MB current + 17-32 MB rotated = ~150 MB/week. No logrotate rule for `seed-*.log`, no Docker `log-opts` max-size. Disk at 63% (45/75 GB). At current rate disk fills in ~4 months.

7. **61 cron entries** in the root crontab. 45 are seed-signal entries (9 timeframes × 5 exchanges, staggered). The cron framework itself is sound — staggering prevents simultaneous fires — but the every-1-minute steady-state is dominated by 5m + 15m seed crons running constantly.

## Postgres Hot-Path Smoking Gun

```
=== sample 5 sec of pg_stat_activity active queries ===
1777548855 active SELECT funding_rate FROM funding_history WHERE coin = $1 AND ...
1777548855 active SELECT funding_rate FROM funding_history WHERE coin = $1 AND ...
1777548855 active SELECT funding_rate FROM funding_history WHERE coin = $1 AND ...
1777548855 active SELECT funding_rate FROM funding_history WHERE coin = $1 AND ...
1777548856 active SELECT funding_rate FROM funding_history WHERE coin = $1 AND ...
1777548856 active SELECT funding_rate FROM funding_history WHERE coin = $1 AND ...
... (7+ concurrent active connections firing the same query shape, sustained)
```

The sole call site: `src/lib/performance-db.ts:636` — `getFundingZScore(coin, currentFunding)`.

```typescript
const cutoff14d = Math.floor(Date.now() / 1000) - 14 * 86400;
const rows = await b.query(
  'SELECT funding_rate FROM funding_history WHERE coin = ? AND recorded_at >= ? ORDER BY recorded_at',
  [coin, cutoff14d]
);
if (rows.length < 20) return null;
// Compute mean, stddev, z-score from rows...
```

`getFundingZScore` is called once per `getTradeSignal()`. With ~250 signals/min from background workloads alone, it executes 250+ DB round-trips per minute. `EXPLAIN ANALYZE` shows the query is fast (0.127 ms execution, hits `idx_funding_coin_time`), but at 250+ queries/min sustained × 100s of ms of postgres-side work-up-and-down (parse, plan, fetch ~30K rows for BTC from a 14d window, transmit) the query churn dominates the box.

| Coin | Rows in funding_history (14d window) |
|---|---|
| BTC  | ~30,000 (from 254,992 lifetime / 22 days × 14) |
| ETH  | ~28,000 |
| SOL  | ~27,000 |
| BNB  | ~26,000 |
| XRP  | ~24,000 |
| DOGE | ~22,000 |
| HYPE | ~5,500 |

Each `getFundingZScore('BTC', x)` call returns ~30K rows over the wire to compute a single z-score. **All in-memory work after the fetch**. A 5-min in-process cache (`coin → zscore` keyed on coin alone) would eliminate >99% of this query traffic with zero correctness risk — 5-minute funding-rate staleness is acceptable for a 5-min-or-larger trade signal, and irrelevant for funding-arb scanning which has its own freshness requirements.

## Decision Answers

### Q1: Can shadow-seed 1m cron safely return to `* * * * *` (every minute) after a no-op deployment?

**Answer: NO without optimizations. CONDITIONAL YES after Optimization #1 (funding cache).**

Justification:
- Current baseline is ~87-99% with 1m cron at `*/2 * * * *`. No headroom for tightening.
- The 5×parallel-process incident at 10:54 UTC showed the box has zero margin for unconstrained process spawning.
- Even with the consolidated single-process pattern, halving the 1m interval from 2 min to 1 min doubles the per-minute cron-time (currently ~12.5s per fire × 30/hr = 375s/hr → 750s/hr). With postgres already at 40%, there's no room.
- **After Optimization #1**: postgres baseline drops to ~12-20%, freeing ~25% of the box. Doubling the 1m sample rate consumes ~5-10% additional CPU. Net new state: 25-40% normalized. Comfortable.

**Recommendation**: ship Optimization #1 first. After 1 hour of measurement at the new baseline, return 1m cron to `* * * * *`. SHADOW_SEED_W1's sample-rate target (3,000 samples/week) is met at every-2-min already; the every-1-min upgrade is "more data faster" not "necessary".

### Q2: Is the CPX31 upgrade (€11.66/mo vs current €5.83/mo, +€7/mo, 4 vCPU / 8 GB RAM) the right move?

**Answer: NO. Optimizations #1 + #2 + #3 + #4 deliver equivalent or greater headroom for €0/mo with 1-2 hours of Code time. CPX31 is a justifiable backstop only if optimizations #1-#2 land and post-optimization measurements still show <20% headroom (predicted: optimizations land + give ~50% headroom).**

Cost-benefit framing:
- **CPX31 upgrade**: +2 vCPU + 4 GB RAM. Bounds future growth at +100% capacity. €84/year recurring. No engineering work needed.
- **Optimizations #1-#4**: ~1-2 hours of Code time, one-time. Frees ~50% of current capacity (equivalent to +1 vCPU on the existing CPX22). €0/year. Compounding: every future workload added benefits from the lower postgres floor.

**The funding-cache optimization is so high-leverage** (50%+ of total CPU baseline removed by ~30 LOC) that paying €84/year to avoid 15 minutes of work is a poor trade.

**Edge case where CPX31 IS the right move**: if the operator's roadmap requires both (a) shadow-seed at every-1-min on the FULL universe (not top-5 restricted), AND (b) GRID_ASSETS expansion to 12+ assets, AND (c) launch of a new Hyperliquid market-making automation that lives on this same box. In that combined scenario, even with optimizations the demand would saturate CPX22. None of those are on the immediate roadmap; revisit Q2 if/when they appear.
